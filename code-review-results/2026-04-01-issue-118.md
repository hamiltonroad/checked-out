## Findings

### HIGH-001: Search query uses Op.like with user input without sanitization

**File:** `backend/src/services/bookService.js` (lines 100, 130)
**Standard:** CLAUDE.md - "Do NOT interpolate user-supplied values into sequelize.literal()"

The `Op.like` usage with `%${search}%` is safe from SQL injection because Sequelize parameterizes Op.like values. However, the `%` and `_` wildcard characters in the user's search input are not escaped, which means a user searching for `%` or `_` would get unintended results. LIKE wildcards in user input should be escaped.

**Recommendation:** Escape LIKE wildcard characters in the search term:
```javascript
const escapedSearch = search.replace(/[%_]/g, '\\$&');
```
Then use `escapedSearch` in the Op.like clauses.

### MEDIUM-001: Duplicate search condition in buildWhereClause

**File:** `backend/src/services/bookService.js` (lines 99-110)
**Standard:** craftsmanship-quick-ref.md - DRY principle

The `if (search)` condition is checked twice in `buildWhereClause` - once to set `bookWhere.title` and again to set `authorWhere`. Since the title filter is later deleted in `getAllBooks` (line 146), setting it in `buildWhereClause` is unnecessary work and confusing.

**Recommendation:** Remove the title assignment from `buildWhereClause` since it's handled via the ID-based search in `getAllBooks`. Only set `authorWhere` in the method.

### LOW-001: BooksPage.test.jsx lost significant test coverage

**File:** `frontend/src/pages/BooksPage.test.jsx`
**Standard:** testing-guide.md - comprehensive test coverage

The original test file had detailed tests for client-side search filtering (by title, author first/last name, partial matches, case-insensitive, debounce timing). Since search is now server-side, the client-side search tests were rightfully removed. However, no new tests were added to verify that the useBooks hook is called with the correct search parameters when the user types in the search box.

**Recommendation:** Consider adding a test that verifies `useBooks` is called with `{ search: 'clean', page: 1, limit: 20 }` after the user types "clean" and the debounce fires. This ensures the wiring between the search input and server-side query is correct.

### LOW-002: Frontend bookService builds URL manually instead of using Axios params

**File:** `frontend/src/services/bookService.js`
**Standard:** frontend-quick-ref.md - API service pattern shows `api.get('/books', { params: filters })`

The service manually builds the URL with `URLSearchParams` instead of using Axios's built-in `params` option, which handles query string construction automatically and is the pattern shown in the frontend quick-ref.

**Recommendation:** Use `api.get('/books', { params })` instead of manual URL construction.
