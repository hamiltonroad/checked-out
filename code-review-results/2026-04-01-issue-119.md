## Findings

### Medium

**M1: CSRF verifyCsrf throws instead of calling next(error)**
- **File:** `backend/src/middlewares/csrf.js`, lines 47-51
- **Issue:** The `verifyCsrf` middleware uses `throw` instead of `next(error)` for error handling. In Express 4, unhandled throws in async-free middleware would crash. In Express 5 (which this project uses), thrown errors in sync middleware are caught, so this works but is inconsistent with the project's `next(error)` pattern used everywhere else.
- **Fix:** Wrap in try/catch or use `next()` pattern for consistency.

**M2: CSRF timing attack on string comparison**
- **File:** `backend/src/middlewares/csrf.js`, line 51
- **Issue:** `cookieToken !== headerToken` uses non-constant-time comparison. For CSRF double-submit, this is low-risk since the attacker doesn't receive timing information from the server, but using `crypto.timingSafeEqual` is best practice for token comparison.
- **Fix:** Use `crypto.timingSafeEqual(Buffer.from(cookieToken), Buffer.from(headerToken))`.

**M3: AuthService.login uses identical error message for "patron not found" and "wrong password"**
- **File:** `backend/src/services/AuthService.js`, lines 24-36
- **Issue:** This is actually good security practice (prevents user enumeration). However, the third case ("Account not set up for password login") leaks information about account state. Consider using the same generic message.
- **Fix:** Change line 29 to use the same "Invalid card number or password" message.

### Low

**L1: LoginPage imports PropTypes but defines empty propTypes**
- **File:** `frontend/src/pages/LoginPage.jsx`, lines 2, 47
- **Issue:** PropTypes import is present but propTypes is empty `{}`. Per project conventions, zero-prop components should still define PropTypes. This is correct behavior per CLAUDE.md, but the import of PropTypes creates an unused-looking import.
- **Recommendation:** Keep as-is; project convention requires it.

**L2: date-fns added as dependency (pre-existing issue)**
- **File:** `frontend/package.json`
- **Issue:** `date-fns` was added to fix a pre-existing build failure (ReviewsList.jsx imports it but it wasn't in dependencies). While necessary for the build to pass, this is technically out of scope.
- **Recommendation:** Keep; the build cannot pass without it.

**L3: CSRF cookie not regenerated after login**
- **File:** `backend/src/middlewares/csrf.js`
- **Issue:** Best practice is to regenerate the CSRF token after authentication state changes (login/logout) to prevent session fixation-style attacks on the CSRF token. Currently, the cookie is only set if missing.
- **Recommendation:** Consider adding CSRF token rotation on login/logout in a follow-up.

**L4: No automatic token refresh on 401 in frontend interceptor**
- **File:** `frontend/src/services/api.js`
- **Issue:** When an access token expires, the frontend does not automatically attempt a refresh. The user would need to be redirected to login. A response interceptor that catches 401 and attempts refresh would improve UX.
- **Recommendation:** Add response interceptor in a follow-up issue.
