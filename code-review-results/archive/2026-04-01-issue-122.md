# Code Review: Issue #122 - Standardize Validation Error Format

**Date:** 2026-04-01
**Reviewer:** code-review-agent
**Branch:** feature/issue-122-standardize-validation-errors

## Summary

This PR standardizes the validation error format across all backend validators to return field-level error objects (`{ field, message }`) instead of flat strings. It also updates the frontend to parse and display these errors, creates an ADR, and refactors the rating validator to use the same middleware pattern as other validators.

## Positive Observations

1. **DRY principle well applied:** The `mapValidationErrors` helper in `validateRequest.js` eliminates the repeated Joi-to-field mapping logic.
2. **Backward compatible:** The top-level `message` field is preserved in all error responses, so consumers that only read `message` still work.
3. **Clean refactoring:** The rating validator was cleanly refactored from inline middleware to the schema-based pattern, reducing duplication.
4. **ADR created:** Architecture decision properly documented following existing ADR format and conventions.
5. **Consistent patterns:** All validators now use the same `validateRequest(validator.action)` pattern.
6. **Frontend utility well-designed:** `parseApiError`, `getFieldError`, and `formatApiError` provide a clean layered API for different use cases.
7. **File size compliance:** All modified files are well under the 200-line limit.

## Overall Assessment

The implementation is clean, follows project standards, maintains backward compatibility, and successfully standardizes the validation error format. The single low-severity finding (unused export) is a design choice for future extensibility, not a defect.
