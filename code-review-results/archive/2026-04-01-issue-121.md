# Code Review: Issue #121 - Standardize on Vitest for Backend and Frontend Testing

**Date:** 2026-04-01
**Reviewer:** code-review-agent
**Branch:** feature/issue-121-standardize-vitest

---

## Summary

This PR migrates the backend test runner from Jest to Vitest, unifying the monorepo on a single test framework. The migration required a novel `require.cache` injection approach to work around Vitest's inability to intercept CJS `require()` calls in source files. An ADR documenting the decision and approach was created.

---

## Findings

### Critical: 0

### High: 0

### Medium: 1

**M1: bookService.test.js exceeds 200-line limit (206 lines)**
- **File:** `backend/src/services/bookService.test.js`
- **Standard:** CLAUDE.md: "Source files must not exceed 200 lines (type/config files exempt)"
- **Details:** Test file is 206 lines, 6 lines over the limit. The extra lines come from the require.cache mock setup boilerplate needed for CJS interop.
- **Recommendation:** Extract the mock setup boilerplate (lines 1-44) into a shared test utility like `backend/src/test-utils/mockModels.js`. This would also reduce duplication across test files (DRY principle).

### Low: 2

**L1: Redundant mock setup pattern across test files**
- **Files:** All 3 test files
- **Standard:** craftsmanship-quick-ref.md: DRY principle
- **Details:** The `createRequire` + `require.cache` injection pattern is repeated in each test file. While each file mocks different modules, the boilerplate structure is identical.
- **Recommendation:** Consider a shared utility function like `injectMock(modulePath, exports)` to reduce boilerplate. This is a minor concern since each test file has different mock shapes.

**L2: `globals: true` in vitest config while also importing from 'vitest'**
- **File:** `backend/vitest.config.js` and all test files
- **Details:** The vitest config sets `globals: true` which makes `describe`, `it`, `expect`, etc. available globally. However, the test files explicitly import these from 'vitest' along with `vi`. The explicit imports are actually the preferred approach (clearer, better IDE support), but the `globals: true` config is then redundant.
- **Recommendation:** Either remove `globals: true` from config (since test files import explicitly), or remove explicit imports for globals and only import `vi`. Low priority since both approaches work.

---

## Overall Assessment

**Recommendation: Pass**

The migration is well-executed given the fundamental CJS interop challenge. The `require.cache` injection approach is pragmatic and well-documented in both code comments and the ADR. All 48 tests pass. The one medium finding (file length) is borderline and the approach is justified by the CJS boilerplate requirement.
