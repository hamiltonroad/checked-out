## Findings

| Severity | File | Issue | Pillar | Resolved |
| -------- | ---- | ----- | ------ | -------- |
| Medium | `frontend/e2e/fixtures/api.ts:64-69` | `primeCsrfFromCookies` duplicates the CSRF priming logic already present in `apiRequest` (the original helper). Both helpers issue `GET /books?limit=1` and pull `_csrf` from cookies. Extract a single shared helper to keep CSRF behavior in lock-step if the backend ever changes the priming endpoint or cookie name. | Architecture | No |
| Medium | `frontend/e2e/security/checkout-body-spoofing.spec.ts:51-74` | The test discovers an available copy by iterating books and copies until `POST /checkouts` returns 201. This is brittle and slow against seed drift, and it pollutes the run if the loop succeeds late (the cleanup only returns the last created checkout — fine here since the loop breaks on first 201, but the structure invites future regressions). The issue's AC explicitly recommends "use a clean copy or seed isolated data via API". Prefer seeding a dedicated copy via an API helper or test-only setup route. | Verification | No |
| Medium | `frontend/e2e/security/checkout-body-spoofing.spec.ts:33` | Magic number `FORGED_PATRON_ID = 999999` is local to the test. Acceptable, but the spec also hardcodes `/books?limit=50` (line 47) — if no available copy exists in the first 50 books on a given seed, the test fails with a confusing message instead of a clear setup error. Either raise/remove the limit or fail fast with an explicit "no available copy in seed" error. | Documentation | No |
| Medium | `frontend/e2e/security/rbac-api-enforcement.spec.ts:28-32` | The POST `/checkouts` assertion uses `copy_id: 1` — a hardcoded id that may not exist or may already be checked out. Because the role check fires before the body is processed, this happens to work, but the test would also pass for the wrong reason if validation ran first and rejected the body with 400. Add a comment explaining the ordering, or use a clearly-invalid sentinel like `copy_id: 0` and assert the status is specifically 403 (not 400). | Verification | No |
| Low | `frontend/e2e/security/checkout-body-spoofing.spec.ts:11-19` | Local interface declarations (`MeData`, `BookListData`, `AvailableCopiesData`, `CheckoutData`) duplicate shapes that almost certainly exist in `frontend/src/types/index.ts`. CLAUDE.md requires shared types to live in one canonical location. E2E tests may legitimately want narrow local shapes, but at minimum import the canonical Book/Copy/Checkout types where they exist. | Documentation | No |
| Low | `frontend/e2e/fixtures/api.ts:97` | `apiRequestRaw` accepts both `ctx` and `page` options but silently prefers `ctx` if both are passed. Document the precedence in the JSDoc or throw if both are supplied. | Documentation | No |
| Low | `frontend/e2e/security/protected-route-redirect.spec.ts:21-23` | Login form interaction is duplicated from `loginAs` in `fixtures/auth.ts`. The test deliberately avoids `loginAs` because it needs the post-login redirect behavior, but a thin `loginAsViaForm(page, role)` helper would remove the duplication and keep label selectors in one place. | Architecture | No |
