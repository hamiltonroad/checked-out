# Code Review: Issue #129 - Tiered Rate Limiting

**Date:** 2026-04-01
**Reviewer:** code-review-agent
**Branch:** feature/issue-129-tiered-rate-limiting
**Commit:** b65d353

---

## Summary

Implementation replaces a global rate limiter with a two-tier system (standard/strict) applied per-route. A new middleware factory module provides configurable rate limiters. ADR-011 documents the decision.

---

## Findings

### Medium

**M1: Rate limit handler bypasses centralized error handling**
- **File:** `backend/src/middlewares/rateLimiter.js` (lines 34-40)
- **Detail:** The custom `handler` in `createRateLimiter` sends a raw JSON response (`res.status(429).json(...)`) instead of using `ApiResponse.error()`. The project convention (per CLAUDE.md and `errorHandler.js`) is to use `ApiResponse.error()` for all API responses.
- **Fix:** Import `ApiResponse` and use `res.status(429).json(ApiResponse.error('Too many requests from this IP, please try again later.', 429))`.

### Low

**L1: Backend quick-ref security section not updated**
- **File:** `standards/quick-ref/backend-quick-ref.md` (line 178)
- **Detail:** The security section still shows the old single global rate limiter pattern. This could mislead future developers referencing the quick-ref.
- **Fix:** Update the security section to reflect tiered rate limiting pattern. (Deferred - documentation update outside implementation scope.)

**L2: Module-level singleton rate limiters share state across tests**
- **File:** `backend/src/middlewares/rateLimiter.js` (lines 43-44)
- **Detail:** `standardLimiter` and `strictLimiter` are created at module load time as singletons. In integration tests, rate limit counters may carry over between test cases. This is acceptable for now since the test suite does not currently test rate limiting directly, but could cause flaky tests in the future.
- **Fix:** No fix needed now. When rate limit tests are added, use the `createRateLimiter` factory to create fresh instances per test.
