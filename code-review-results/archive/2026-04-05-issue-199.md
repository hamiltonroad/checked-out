## Findings

### Medium

#### M1: WishlistPage BookCard book prop may lack required fields
- **File:** `frontend/src/pages/WishlistPage.tsx` line 69
- **Description:** The `entry.book` object from the wishlist API only includes `['id', 'title', 'isbn', 'genre', 'has_profanity']` with authors. BookCard also renders `copies` (via `getCopyCountText`) and `average_rating`/`total_ratings` (via RatingDisplay). These fields won't be in the wishlist response, so BookCard will render "No copies" and skip rating display. This is acceptable for MVP but could confuse users.
- **Impact:** Visual inconsistency on wishlist page vs books page
- **Recommendation:** Consider including copies and rating stats in the wishlist API response, or accept the simpler display for wishlist cards.
- **Resolved:** No (deferred — acceptable for MVP, not a bug)

#### M2: WishlistPage uses non-standard navigation pattern for book click
- **File:** `frontend/src/pages/WishlistPage.tsx` line 24
- **Description:** `handleBookClick` navigates to `/?book=${bookId}` but BooksPage doesn't read a `book` query parameter — it opens a modal based on local state. This navigation won't actually open the book detail.
- **Impact:** Clicking a book card on the wishlist page navigates to books page but doesn't open the detail modal.
- **Recommendation:** Navigate to `/` and let the user find the book, or implement query parameter handling in BooksPage.
- **Resolved:** Yes — will fix

### Low

#### L1: Optimistic update creates temporary entry with id: -1
- **File:** `frontend/src/hooks/useWishlist.ts` line 32
- **Description:** The optimistic update in `useAddToWishlist` creates a temporary entry with `id: -1` and no `book` property (type assertion used). If the component tries to access `entry.book` properties before `onSettled` completes, it could cause runtime errors.
- **Impact:** Brief flash of undefined book data during optimistic update (mitigated by React Query invalidation on settle).
- **Recommendation:** Either include a placeholder book object or handle missing book data gracefully.
- **Resolved:** No (deferred — low risk due to quick invalidation)

#### L2: DELETE endpoint uses request body instead of URL params
- **File:** `backend/src/routes/wishlistRoutes.js`, `frontend/src/services/wishlistService.ts`
- **Description:** The DELETE /wishlists endpoint receives `book_id` in the request body rather than URL path. While functional, some HTTP clients and proxies may strip request bodies from DELETE requests.
- **Impact:** Low — works with axios and Express, but non-standard REST pattern.
- **Recommendation:** Could change to DELETE /wishlists/:bookId but current approach matches the issue spec.
- **Resolved:** No (deferred — matches issue spec)
