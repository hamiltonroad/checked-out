## Findings

### Medium

#### M1: setTimeout timer not cleared on successful authenticate (HealthService.js:21-26)

The `Promise.race` pattern creates a `setTimeout` that is never cleared when `sequelize.authenticate()` resolves first. While the rejected promise from the timer is harmless (no unhandled rejection since the race is already settled), the timer continues to exist in memory until it fires. Under frequent health checks this is a minor resource inefficiency.

**Fix:** Use `AbortController` or clear the timer after the race resolves.

**File:** `backend/src/services/HealthService.js` lines 21-26

---

### Low

#### L1: Database error message exposed in health response (HealthService.js:40)

The `error` field in the returned object includes the raw error message from the database driver. While health endpoints are typically internal, the error message could leak implementation details (e.g., hostname, port, driver version) to callers.

**Recommendation:** Consider omitting the `error` field from the service return value, or sanitizing it. The controller does not currently include it in the HTTP response, so this is defense-in-depth only.

**File:** `backend/src/services/HealthService.js` line 40
