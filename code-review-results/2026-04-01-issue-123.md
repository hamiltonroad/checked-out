# Code Review: Issue #123 — Route Error Responses Through ApiResponse

**Date:** 2026-04-01
**Reviewer:** code-review-agent
**Branch:** feature/issue-123-route-errors-apiresponse

## Summary

This PR updates the error handler middleware to use `ApiResponse.error()` instead of building raw JSON, adds an `extras` parameter to `ApiResponse.error()` for development-only fields, creates ADR-022, and updates the ADR index.

### Medium: 1

**M1: backend-quick-ref.md shows outdated errorHandler pattern**
- **File:** `standards/quick-ref/backend-quick-ref.md` (lines 98-106)
- **Issue:** The quick reference still shows the old hand-built JSON error handler pattern. Since this PR standardizes error responses through `ApiResponse.error()`, the quick-ref should be updated to reflect the new pattern.
- **Recommendation:** Update the error handling code snippet in `backend-quick-ref.md` to use `ApiResponse.error()`.

### Low: 1

**L1: No unit test for ApiResponse.error() with extras parameter**
- **File:** `backend/src/utils/ApiResponse.js`
- **Issue:** The new `extras` parameter on `ApiResponse.error()` is not covered by a unit test. While the change is simple, a test would prevent regressions.
- **Recommendation:** Add a test verifying `ApiResponse.error('msg', 500, { stack: 'trace' })` includes the stack field.

## Overall Assessment

**Recommendation: Pass with fixes**

The implementation is clean, backward-compatible, and follows project conventions. The core changes are correct and well-structured. One medium finding (outdated quick-ref docs) should be fixed before merge. The low finding (missing test) is deferred as optional.
