## Findings

| Severity | File | Issue | Pillar | Resolved |
| -------- | ---- | ----- | ------ | -------- |
| High | `frontend/src/pages/WishlistPage.tsx:24-29` | `useEffect` depends on the entire `removeFromWishlist` mutation object instead of `removeFromWishlist.isSuccess`. The mutation object identity changes on every render, causing the effect to fire continuously and call `reset()` repeatedly. Fix: depend on `removeFromWishlist.isSuccess` and gate the body on it. | Verification | No |
| High | `frontend/src/components/CurrentCheckoutsTab/CurrentCheckoutsTab.tsx:36-41` and `frontend/src/components/OverdueCheckoutList/OverdueCheckoutList.tsx:35-40` | Loading state clears only when the checkout row disappears from the list (mutation success). On mutation failure the row remains, so `returningId` stays set, the ConfirmDialog stays in `loading=true` forever, and the user is locked out of cancelling or retrying. The error path needs explicit handling (parent should signal failure, or use `useMutation`'s state directly). | Verification | No |
| Medium | `frontend/src/components/WaitlistCard/WaitlistCard.tsx:25-32` | The `confirmed` flag closes the dialog when `isLeaving` flips false, but if the mutation errors the parent still flips `isLeaving` back to false — the dialog will silently close as if it succeeded, hiding the failure from the user. Tie dialog state to mutation status (success vs error) explicitly. | Verification | No |
| Medium | `frontend/src/components/CurrentCheckoutsTab/CurrentCheckoutsTab.tsx`, `frontend/src/components/OverdueCheckoutList/OverdueCheckoutList.tsx`, `frontend/src/components/WaitlistCard/WaitlistCard.tsx`, `frontend/src/pages/WishlistPage.tsx` | The "open confirm → track returningId → effect to clear on completion" state machine is duplicated across four components with subtle variations. Extract into a `useConfirmAction` hook (or have `ConfirmDialog` accept a mutation object) to eliminate copy-paste and centralize the success/error semantics. | Architecture | No |
| Medium | `frontend/src/components/ConfirmDialog/ConfirmDialog.tsx:60` | `Dialog`'s `onClose` is set to `undefined` when loading. MUI still treats backdrop/escape as a close attempt and the prop type expects a function — passing `undefined` works but is non-idiomatic. Prefer `disableEscapeKeyDown` plus a no-op `onClose`, or document the intent. | Documentation | No |

Consider extracting the row into its own component before the next change pushes it over. | Architecture | No |
| Low | `frontend/src/components/HoldCard/HoldCard.tsx:109-134` | The Tooltip+disabled-Button pattern is duplicated verbatim with the existing `ConfirmDialog`'s confirm-button wrapper. Could share a `<LoadingButton>` helper or adopt MUI Lab `LoadingButton`. | Architecture | No |

## Harness Improvement Recommendations

| Finding | Harness Recommendation | Harness Change Made |
| ------- | ---------------------- | ------------------- |
| useEffect depending on mutation object | Add an ESLint rule (or extend `react-hooks/exhaustive-deps` config) flagging React Query mutation objects in effect dependency arrays — they are unstable references. Document in `frontend-quick-ref.md`. | No |
| Mutation error leaves dialog stuck loading | Add a CLAUDE.md constraint: "Confirm-dialog loading state MUST be derived from mutation status (`isPending`/`isError`/`isSuccess`), not from list-membership inference. Mutation errors must close or surface in the dialog." | No |
| Duplicated confirm-action state machine | Codify a `useConfirmAction(mutation)` hook in `frontend/src/hooks/` and reference it from the UX standards doc; reviewers should reject ad-hoc duplications. | No |
| Silent dialog close on mutation failure | Same as above — derive from mutation status, not parent boolean toggles. | No |
| ConfirmDialog `onClose=undefined` | Add to component standards: prefer `disableEscapeKeyDown` over `onClose={undefined}` for non-idiomatic loading guards. | No |
| OverdueCheckoutList row button missing confirm-open guard | Frontend quick-ref should call out: "When opening a confirm dialog from a list row, the trigger must also be disabled while the dialog is open." | No |
| File approaching 200-line cap | The existing `max-lines` ESLint rule already enforces this — no harness change needed; flagged as Boy Scout follow-up. | No |
