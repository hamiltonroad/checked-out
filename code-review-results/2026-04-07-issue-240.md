## Findings

| Severity | File | Issue | Pillar | Resolved |
| -------- | ---- | ----- | ------ | -------- |
| Medium | `backend/src/services/CopyService.js:74-90` | `findCheckoutable` performs an in-memory filter then `slice(limit)` after fetching ALL copies matching the where clause. With a growing catalog this scans the entire `copies` table on every call. Push waitlist gating into the SQL query (LEFT JOIN / NOT EXISTS) and apply `limit` at the DB level, or at minimum pass `limit` to `Copy.findAll` and post-filter only the returned page. | Architecture | No |
| Medium | `backend/src/services/CopyService.js:84` | Waitlist gating filters only on `(book_id, format)` pairs of the front-of-line waiter, but `Copy.findAll` is not constrained to those pairs and the `WaitlistEntry.findAll` runs unscoped when `bookId`/`format` are absent. On a populated DB this returns the full waitlist head set on every call. Same fix as above (single SQL query with NOT EXISTS) eliminates the second round trip. | Architecture | No |
| Medium | `backend/src/validators/copyValidator.js:13` and `backend/src/services/CopyService.js:4` | Default limit `10` is duplicated: Joi `.default(10)` in the validator and `DEFAULT_CHECKOUTABLE_LIMIT = 10` in the service. Drift risk — if one is bumped the other silently lies. Import the service constant into the validator (or vice versa) so there is one source of truth. | Documentation | No |
| Medium | `backend/src/middlewares/rateLimiter.js:50-58` | No automated test asserts the TEST_MODE branch — neither that limiters become no-ops when `TEST_MODE=true` nor that they remain active when unset. Given the security weight of this flag (server can be tricked into running wide-open), a unit test is warranted. | Verification | No |
